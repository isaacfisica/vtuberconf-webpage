# 버튜버학회 (VTuberConf) — Agent Guide

A statically-exported Next.js site for a fictional academic society of virtual streamers
("버튜버학회"). It renders a home page, an about page, a member directory, and per-member
profile pages, deployed to GitHub Pages.

- **Framework:** Next.js 16 (App Router) + React 19, TypeScript (strict).
- **Output:** `output: "export"` — a fully static site. No server, no API routes, no
  runtime data fetching. Everything is generated at build time.
- **Styling:** Plain CSS with a design-token system in `src/app/globals.css`. No CSS
  framework, no CSS-in-JS library. Components use inline `style` props + CSS variables.
- **Content:** All member data lives as TypeScript modules under `src/lib/members/`.

## Commands

```bash
npm run dev     # runs gen-members, then `next dev`
npm run build   # runs gen-members, then `next build` → static export into ./out
npm run lint    # eslint (flat config in eslint.config.mjs)
```

There is no test suite. Validate changes with `npm run build` (catches type + export errors).

## Architecture

```
src/
  app/                     # App Router routes (server components)
    layout.tsx             # root layout: TopBar + main + SiteFooter, <html lang="ko">
    page.tsx               # "/"            → HomePage
    about/page.tsx         # "/about"       → AboutPage
    directory/page.tsx     # "/directory"   → DirectoryPage
    members/[id]/page.tsx  # "/members/:id" → ProfilePage (generateStaticParams over MEMBERS)
    globals.css            # design tokens + base styles
  components/              # presentational components ('use client' where interactive)
    ui/                    # reusable primitives: Avatar, Badge, Button, Card, Input, Tabs, Tag
  lib/
    data.ts                # re-exports MEMBERS + Member type; decorateMember, FIELDS, STATS, etc.
    types.ts               # Member, SnsLink types
    members/               # one file per member (e.g. isaac.ts), default-exporting a Member
    members/index.ts       # GENERATED — do not edit by hand
    basePath.ts            # BASE_PATH + asset() helper for static assets
    navbar.tsx, footer.tsx, blocks.tsx, cosmetics.tsx
scripts/gen-members.mjs    # codegen for members/index.ts
```

- **Routes are thin.** `app/**/page.tsx` files just load data and delegate to a component
  in `src/components/`. Put real UI/logic in `src/components`, not in route files.
- **Path alias:** `@/*` → `src/*` (see `tsconfig.json`).

## Member data (read this before editing members)

Each member is a `Member` (see `src/lib/types.ts`) default-exported from its own file in
`src/lib/members/`. To add a member, create `src/lib/members/<id>.ts`:

```ts
import type { Member } from '../types';

const newmember: Member = {
  id: 'newmember',          // must match filename and is the route segment /members/<id>
  name: '이름',
  field: '전공 분야',
  degree: '박사과정',
  bio: '짧은 소개',
  bioLong: '긴 소개',
  // optional: tags, accent, joined, role, live, sns, channel, deg, handle
};

export default newmember;
```

**`src/lib/members/index.ts` is generated by `scripts/gen-members.mjs`.** It scans the
directory, imports every `*.ts` (except `index.ts`), and builds the `MEMBERS` array. The
script runs automatically before `dev` and `build`. Never hand-edit `index.ts`; just add/
remove member files and let codegen handle the wiring.

`decorateMember()` in `data.ts` augments a member at render time (first name, SNS glyphs/
colors). `FIELDS`, `PRINCIPLES`, `STATS`, `FOOTER_COLS` are static page content also in
`data.ts`.

## basePath / GitHub Pages — easy to get wrong

The site is served from a sub-path on GitHub Pages (`/vtuberconf-webpage/`), configured via
`basePath`/`assetPrefix` in `next.config.ts` and the `NEXT_PUBLIC_BASE_PATH` env var.

- `next/link` and the router apply basePath **automatically**.
- `next/image` (and any raw `<img src>` / asset URL string) does **not**. Prefix those with
  `asset()` from `src/lib/basePath.ts`, e.g. `asset('/assets/seal.svg')`.
- Keep `basePath` in `next.config.ts` and `BASE_PATH` in `basePath.ts` consistent (both
  ultimately read `NEXT_PUBLIC_BASE_PATH`).

## Styling conventions

- Use the CSS custom properties defined in `src/app/globals.css` (ink/parchment/oxblood/
  brass/violet/cyan palettes + semantic tokens like `--text-primary`, `--surface-page`,
  `--font-sans`). Don't hardcode hex colors that duplicate an existing token.
- Components style with inline `style={{ ... }}` referencing `var(--token)`. Match the
  surrounding style; there is no utility-class system.

## Deployment

`.github/workflows/nextjs.yml` builds on push to `master` and deploys `./out` to GitHub
Pages. The workflow sets `NEXT_PUBLIC_BASE_PATH` (and `NEXT_PUBLIC_GA_ID`) at build time.
Because of `output: "export"`, anything requiring a Node server at runtime will not work.

## Gotchas

- Static export only: no `next/headers`, no dynamic `cookies()`, no route handlers, no ISR.
- `generateStaticParams` in `members/[id]/page.tsx` must cover every member for the page to
  be exported — it derives from `MEMBERS`, so adding a member file is sufficient.
- `params` is a `Promise` in route components — `await` it (see `members/[id]/page.tsx`).
- The repo is Korean-language (UI copy, member bios). Keep `lang="ko"` and copy in Korean.
